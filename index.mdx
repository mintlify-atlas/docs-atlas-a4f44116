Harden-Runner is a CI/CD security agent that works like an EDR (Endpoint Detection & Response) for GitHub Actions runners. It monitors outbound network traffic, file system changes, and process activity — correlating every event to the exact workflow step that caused it.

Trusted by over **11,000 open-source projects and enterprises**, including Microsoft, Google, CISA, and Kubernetes, Harden-Runner secures more than **18 million CI/CD workflow runs every week**.

## Why Harden-Runner?

CI/CD runners have access to your most sensitive secrets — cloud credentials, signing keys, deployment tokens. Yet they're typically ephemeral, unmonitored, and invisible to traditional security tools.

Harden-Runner fills this gap:

- **Block exfiltration**: Restrict outbound traffic to only the domains your workflow needs
- **Detect tampering**: Catch unauthorized source code or binary modifications during builds
- **Spot anomalies**: Identify unexpected network calls, new processes, or unusual file writes
- **Minimize permissions**: Determine the minimum `GITHUB_TOKEN` permissions your workflow actually uses

## Real-world detections

Harden-Runner has detected real supply chain attacks in the wild:

- **[tj-actions/changed-files compromise](https://www.stepsecurity.io/blog/harden-runner-detection-tj-actions-changed-files-action-is-compromised)** (CVE-2025-30066) — detected before widespread impact
- **NX Build System malware** — caught data-stealing code in a popular build tool
- **CNCF Backstage Sha1-Hulud attack** — identified anomalous network behavior in real time
- **Google Flank CI/CD attack** — correlated suspicious traffic to the compromised step

<Note>
  Harden-Runner is open source (Apache 2.0) for public repositories. Enterprise features — including private repo support, self-hosted runners, and GitHub Checks integration — are available with a [StepSecurity subscription](https://www.stepsecurity.io/pricing).
</Note>

## Get started in 2 minutes

<Steps>
  <Step title="Add Harden-Runner to your workflow">
    Insert the action as the **first step** in any job:

    ```yaml .github/workflows/your-workflow.yml
    steps:
      - name: Harden Runner
        uses: step-security/harden-runner@v2
        with:
          egress-policy: audit
    ```
  </Step>
  <Step title="Run your workflow">
    Push your changes and let the workflow run. Harden-Runner installs a lightweight agent on the runner and begins monitoring.
  </Step>
  <Step title="Review security insights">
    After the job completes, a link appears in the workflow logs and job summary. Click it to see every outbound network call, file write, and process execution — mapped to the exact step that triggered it.
  </Step>
  <Step title="Switch to block mode">
    Once you've reviewed your baseline, update your workflow to use `egress-policy: block` with an `allowed-endpoints` list to prevent unauthorized traffic.
  </Step>
</Steps>

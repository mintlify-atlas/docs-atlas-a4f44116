This guide walks you through adding Harden-Runner to an existing workflow, reviewing the security insights it produces, and switching from monitoring-only (audit) mode to active enforcement (block) mode.

## Prerequisites

- A GitHub repository with at least one GitHub Actions workflow
- Permission to edit workflow files in the repository

<Tabs>
  <Tab title="Linux">
    Linux runners have full support: network egress monitoring, file monitoring, process monitoring, and block mode.
  </Tab>
  <Tab title="Windows">
    Windows runners support audit mode. Network events are captured and visible in the security insights dashboard.
  </Tab>
  <Tab title="macOS">
    macOS runners support audit mode. The agent is installed via a platform-specific installer package.
  </Tab>
</Tabs>

## Setup

<Steps>
  <Step title="Add Harden-Runner as the first step in your job">
    Open your workflow file (`.github/workflows/<your-workflow>.yml`) and add Harden-Runner as the **first step** inside the job you want to monitor. Starting with `egress-policy: audit` is recommended — it records all outbound traffic without blocking anything.

    ```yaml
    jobs:
      build:
        runs-on: ubuntu-latest
        steps:
          - name: Harden Runner
            uses: step-security/harden-runner@v2
            with:
              egress-policy: audit

          - name: Checkout
            uses: actions/checkout@v4

          # ... rest of your steps
    ```

    <Tip>
      You can automate this step by pasting your workflow YAML into the [StepSecurity secure-workflow tool](https://app.stepsecurity.io/secure-workflow). It adds Harden-Runner to every job and generates an `allowed-endpoints` list based on the actions you use.
    </Tip>
  </Step>

  <Step title="Run the workflow and find the insights link">
    Commit and push your change to trigger the workflow. Once the job completes:

    1. Open the workflow run in GitHub Actions.
    2. Expand the **Harden Runner** step logs.
    3. Look for a line that reads **"View security insights and recommended policy at:"** followed by a URL to `app.stepsecurity.io`.

    The same link also appears in the **job summary** at the bottom of the workflow run page.
  </Step>

  <Step title="Review the security insights">
    Click the link to open the security insights dashboard. You will see:

    - **Network events** — every outbound network connection made during the job, correlated to the specific step that made it, including the domain, IP address, process name, and PID
    - **File events** — file writes tracked during the job (enterprise tier)
    - **Process events** — processes executed and their arguments (enterprise tier)

    Review the network events to understand what your workflow actually calls. This becomes the basis for your allowed-endpoints list.

    <Note>
      For public repositories, insights are available immediately with no account required. For private repositories, a [StepSecurity subscription](https://www.stepsecurity.io/pricing) is required to enable monitoring.
    </Note>
  </Step>

  <Step title="Switch to block mode with an allowed-endpoints list">
    Once you understand your workflow's network behavior, switch from `audit` to `block` and add an `allowed-endpoints` list. This prevents any outbound connection not on the list from succeeding.

    ```yaml
    jobs:
      build:
        runs-on: ubuntu-latest
        steps:
          - name: Harden Runner
            uses: step-security/harden-runner@v2
            with:
              egress-policy: block
              allowed-endpoints: >
                github.com:443
                api.github.com:443
                objects.githubusercontent.com:443
                registry.npmjs.org:443

          - name: Checkout
            uses: actions/checkout@v4

          - name: Install dependencies
            run: npm ci

          # ... rest of your steps
    ```

    The `allowed-endpoints` field accepts a space- or newline-separated list of `hostname:port` entries. Ports are required.

    <Warning>
      If you set `egress-policy: block` with an empty `allowed-endpoints` list, **no outbound traffic will be allowed** for any step in the job. This will break most workflows. Always review audit mode results first and build your allowed-endpoints list before switching to block mode.
    </Warning>
  </Step>
</Steps>

## Common allowed-endpoints patterns

The endpoints your workflow needs depend on what it does, but most workflows use some combination of the following:

```yaml
with:
  egress-policy: block
  allowed-endpoints: >
    github.com:443
    api.github.com:443
    objects.githubusercontent.com:443
    codeload.github.com:443
    uploads.github.com:443
    registry.npmjs.org:443
    pypi.org:443
    files.pythonhosted.org:443
    proxy.golang.org:443
    sum.golang.org:443
    storage.googleapis.com:443
```

The security insights dashboard shows the exact domains your workflow calls, making it straightforward to build this list from observed behavior rather than guessing.

## Using a policy from the policy store

Instead of configuring inputs directly in each workflow, you can store a named policy in the StepSecurity API and reference it by name:

```yaml
- name: Harden Runner
  uses: step-security/harden-runner@v2
  with:
    policy: my-org-policy
```

<Note>
  Using the `policy` input requires `id-token: write` permission in your job, as it uses OIDC to fetch the policy from the StepSecurity API.
</Note>
